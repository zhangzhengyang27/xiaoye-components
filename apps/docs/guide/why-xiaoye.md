# 设计理念

`xiaoye-components` 不以“大而全替代者”为目标，而是优先把中后台页面里的高频路径做顺手、做稳定、做成文档能直接复用的形态。

## 为什么做这个项目

很多中小团队在做后台系统时，真正卡住的不是“有没有组件”，而是下面这些问题：

- 同类组件在命名和交互上不统一，切换时要不断重新记忆。
- 表单、弹层、表格能单独工作，但放到同一页里以后细节开始打架。
- 文档能看到静态效果，却很难快速拼出一个完整页面。

这个项目想解决的，就是这些更贴近日常开发的问题。

## 当前优先解决什么

### 低心智负担的 API

组件之间尽量共用一致的状态语义、尺寸约定和命名方式，减少“这个组件为什么又不一样”的切换成本。

### 页面级闭环

文档不会只停留在“单个组件长什么样”，而是尽量让 `Button / Form / Modal / Table / Pagination` 这些组件能在一个真实页面里串起来。

### 中文优先的示例

这套文档不做国际化分流，所有术语、文案和页面语境都保持中文，让团队内部讨论和复制示例更直接。

## 适合什么项目

- 后台管理系统
- SaaS 管理台
- 中小型业务平台
- 需要快速交付的内部工具项目

## 当前不打算解决什么

- 追求极其完整组件覆盖的大型平台
- 重度品牌定制或复杂设计系统落地
- 低代码、虚拟表格、复杂树表、日期生态为主的系统

:::warning 当前边界
如果你的项目核心需求是“组件种类必须非常全”，那么 `xiaoye-components` 当前阶段并不适合直接替代成熟的大型组件库。
:::

## 文档写作约定

为了让文档真正服务开发，而不是只做展示页，这个站点会尽量遵循下面三条约定：

- 先给示例，再解释适用场景，最后补 API 摘要。
- 优先用真实页面语境描述组件，而不是只写抽象属性名。
- 聚焦当前版本已经稳定可用的能力，不提前承诺未来尚未落地的功能。
