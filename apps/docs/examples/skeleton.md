# Skeleton 场景示例

这页不再重复 `xy-skeleton` 的基础 API，而是直接把它放回一个控制台混合页里，观察骨架屏如何同时承接指标卡、任务队列和摘要区块。组件定位和完整属性说明仍然放在 [Skeleton 骨架屏](/components/skeleton) 页面。

:::tip 阅读方式
先看顶部指标卡的骨架，再看中间任务队列，最后看右侧摘要块。这样更容易判断你的页面到底需要默认骨架、`SkeletonItem` 组合，还是两者一起用。
:::

## 推荐阅读路径

1. 先看顶部概览卡，理解骨架屏如何承接控制台常见的 KPI 区块。
2. 再看中间任务队列，判断列表区更适合默认模板还是自定义 `template`。
3. 最后看右侧摘要块，确认详情区和补充说明如何稳定占位。

## 适合什么

- 控制台首页、发布中心、任务中心这类“页面结构固定，但接口返回较慢”的后台页面。
- 同一屏里既有指标卡、又有列表区和右侧摘要区，需要统一 loading 反馈的场景。
- 想让用户先看见页面结构，而不是只看到一个通用 loading 图标的页面。

## 不适合什么

- 需要表达步骤依赖、流程编排或时间轴关系的复杂流程型页面。
- 页面结构完全未知、只能先展示通用 loading 的场景。
- 已经有稳定的缓存数据，不需要再额外渲染占位反馈的区域。

## 数据流说明

1. 页面层维护真实 `loading` 状态，并决定哪些区块仍在等待数据。
2. `xy-skeleton` 只负责承接占位反馈，不处理请求和数据同步逻辑。
3. 当 `loading` 切到 `false` 时，默认插槽直接回到真实内容区，不再重复维护两份页面结构。

## 从这个页面可以直接复用什么

1. 顶部概览卡的骨架结构，适合大多数控制台首页。
2. 列表 / 表格区的批量占位模板，适合任务中心和发布队列。
3. 右侧摘要块的按钮 + 文本骨架，适合详情抽屉、说明区和推荐动作面板。
