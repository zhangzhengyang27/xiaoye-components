# xiaoye-components 组件库规划方案

## 1. 项目判断

这个项目最适合走的不是单一类型，而是**混合型组件库**，但主路径应明确为：

**个人品牌型 + 开源型 + 轻业务型验证**。

说明该项目最适合走哪条路径，以及原因。

- 最适合的主路径：`个人品牌型开源组件库`
- 次级属性：`轻业务型`
- 不建议的路径：`一开始就做大而全通用组件库`

原因很直接：

- 你只有 1 个人，3 个月做 MVP，这与“完整对标 Element Plus / Ant Design Vue”天然冲突。
- 真正的大而全组件库，难点不在于把组件“写出来”，而在于长期兼容、API 治理、文档维护、问题处理、边界场景兜底、版本策略和生态心智建设。
- 个人开发者如果一开始就把目标设成“通用大而全”，通常会死在中后期维护，而不是死在初期开发。

更现实的目标应该是：

- 先做一个**面向中后台场景的高频组件子集**
- 用**易用性 + 类型体验 + 文档体验**打出差异化
- 在 3 个月内发布一个可安装、可试用、可反馈、可迭代的 `v0.x`
- 让别人形成一个清晰认知：  
  **“这是一个专为中后台开发者做的、上手顺手、类型友好、文档清楚的 Vue 3 组件库。”**

结论先行：

- 这个项目不适合以“替代 Element Plus / Ant Design Vue”为目标
- 适合以“在中后台高频场景里提供更轻、更顺手、更清晰的开发体验”为目标
- 第一阶段的胜利标准不是组件数量，而是**用户愿意试、愿意留、愿意提反馈**

## 2. 关键假设

列出你采用的关键假设。

- 仓库当前确实为空目录，且尚未初始化 Git 仓库。
- 本地环境可用：`Node v24.14.0`、`pnpm v10.30.3`。
- 前 6-12 个月的首要目标是：`开源认知`。
- MVP 的主卖点优先级是：`易用性`。
- 你接受首个公开版本是 `0.x`，不是“生产级全覆盖”。
- 首批组件中 `Select`、`Form`、`Table` 会做，但只做**第一阶段可控能力**，不追求全功能。
- 首期不会把暗黑模式、低代码、完整国际化、完整 SSR 优化当成主战场。
- 你愿意引入少量成熟依赖解决底层通用问题，而不是所有能力都自己重造。
- 项目定位是“长期开源资产”，因此文档、命名、类型、版本发布必须从第一天就有基本规范。

## 3. 机会与风险分析

| 维度           | 机会                                                      | 风险                                                        | 建议                                               |
| -------------- | --------------------------------------------------------- | ----------------------------------------------------------- | -------------------------------------------------- |
| 市场现实       | Vue 中后台场景长期存在，组件库仍有稳定需求                | Element Plus、Ant Design Vue 已经非常成熟，通用替代机会很小 | 不做“替代者”，做“更聚焦的 DX 方案”                 |
| 目标用户       | 中后台开发者非常看重交付效率、默认行为、表单表格体验      | 他们对迁移成本敏感，不会轻易切换整套组件库                  | 从新项目、个人项目、小团队项目切入，不强打存量替换 |
| 产品空间       | 易用性、类型体验、文档体验仍有提升空间                    | 如果只是换皮和重命名，会被视为重复造轮子                    | 必须有明确价值主张，不要只做“另一个 Element Plus”  |
| 工程实现       | Vue 3 + TS + Vite + pnpm 生态成熟，起步成本低             | 表格、表单、弹层、选择器都属于高复杂度组件                  | MVP 先做窄范围功能，不要一上来追求全量特性         |
| 开源传播       | 个人品牌项目天然适合内容传播、案例输出、技术沉淀          | 如果没有 Demo、文档、真实场景，很难获得第一批用户           | 文档站、示例站、管理后台样板必须尽早出现           |
| 差异化         | 类型提示、默认配置、 API 一致性、复制即用案例是高感知价值 | 这些工作“看起来不酷”，但非常耗时                            | 把“看不见但有价值”的体验做成核心竞争力             |
| 维护模型       | 一人项目更容易统一审美和技术路线                          | 一旦范围过大，issue、兼容和回归会迅速压垮维护者             | 严格限制范围，优先高频组件和高复用场景             |
| 商业/品牌价值  | 可成为个人技术名片、招聘和合作资产                        | 如果长期没人用，容易变成单纯技术自嗨                        | 用真实业务 Demo 和公开路线图验证方向               |
| 容易高估的地方 | 自己的开发速度、首版组件数量、用户迁移意愿                | 组件写完不等于可用，真正成本在边界和维护                    | 把“可维护”视为第一约束                             |
| 容易低估的地方 | 文档、示例、类型测试、a11y、发布和回归成本                | 这些不做，项目就很难进入“可用”状态                          | 工程基础必须早做，哪怕先做最小版                   |

明确判断：

- 这是一个**值得投入**的方向，但前提是目标正确。
- 最推荐的切入方式：  
  **“中后台高频组件 + 更好的易用性 + 更清晰的文档与类型体验”**
- 最不建议的切入方式：  
  **“从 Day 1 就做大而全、全场景、全主题、全生态替代型组件库”**

## 4. 产品定位

请包含：

- 一句话定位
- 目标用户
- 核心场景
- 差异化方向
- 不做什么

**一句话定位**

`xiaoye-components` 是一个面向中后台系统的 Vue 3 组件库，主打低心智负担的 API、清晰的类型体验和可直接落地的文档案例。

**目标用户**

- 使用 Vue 3 搭建后台管理系统、SaaS 系统、业务平台的前端开发者
- 小团队或个人开发者
- 对交付效率、API 一致性、文档可读性比“绝对功能数量”更敏感的人群
- 新项目启动者，而非重度历史包袱项目维护者

**核心场景**

- 登录后管理后台
- CRUD 列表页
- 搜索筛选 + 表单录入
- 详情页 / 弹窗编辑 / 标签切换
- 中等复杂度业务系统，而不是强定制 BI、复杂编辑器、低代码平台

**相比 Element Plus / Ant Design Vue 的差异化方向**

- 不拼“大而全”，拼“常用场景更顺手”
- 不拼“组件数量”，拼“默认行为更合理”
- 不拼“复杂设计规范体系”，拼“开发者接入和理解成本更低”
- 不拼“全行业覆盖”，拼“中后台高频场景体验更直接”
- 把“类型提示、示例质量、复制即用”做成第一阶段品牌感知点

**明确不做什么**

- 不做“Element Plus / Ant Design Vue 的完整平替”
- 不做首期低代码、表单配置平台、Pro Components 套件
- 不做首期完整暗黑模式和多品牌主题系统
- 不做首期重型 DataGrid、虚拟滚动表格、树表、拖拽布局系统
- 不做首期日期生态全家桶
- 不做首期自研整套图标库

**MVP 应聚焦的能力**

- 高频基础组件
- 高频表单交互
- 基础表格展示
- 一致化 API
- 类型可感知
- 文档站可快速复制使用
- 一个真实的后台示例页或 Mini Admin Demo

**第一阶段最应该建立的品牌印象**

- “这个库不花哨，但特别顺手”
- “文档清楚，拿来就能用”
- “API 命名统一，不用猜”
- “作为个人项目，完成度 surprisingly 高”

## 5. 差异化策略

先给结论：

- 最推荐的主差异化方向：**易用性**
- 2 个可搭配的副差异化方向：**类型体验**、**文档体验**
- 3 个前期不值得投入的方向：**深度暗黑/多品牌主题**、**低代码/配置化表单体系**、**重型高性能组件能力**

| 方向                | 是否推荐   | 价值 | 成本 | 适合当前阶段吗   | 说明                                                                                  |
| ------------------- | ---------- | ---- | ---- | ---------------- | ------------------------------------------------------------------------------------- |
| API 设计风格        | 强烈推荐   | 极高 | 中   | 是               | 统一 `size / variant / status / disabled / loading / modelValue` 等语义，减少学习成本 |
| 使用门槛和易用性    | 主推荐     | 极高 | 中   | 是               | 默认值合理、组合顺手、低配置可用，是最容易被感知的价值                                |
| TypeScript 类型体验 | 推荐       | 高   | 中   | 是               | 提升 props、emits、slots、表单规则、表格列配置体验，形成专业感                        |
| 文档站体验          | 推荐       | 高   | 中   | 是               | 对个人开源项目来说，文档几乎就是产品本体的一部分                                      |
| 样式定制与主题系统  | 推荐但收敛 | 中高 | 中   | 是，但只做基础版 | 首期只做 CSS 变量 + Token，不做复杂主题生成器                                         |
| 表单场景支持        | 推荐       | 高   | 高   | 是，但范围要窄   | `Form + Input + Select + Modal` 是中后台使用频次最高的组合之一                        |
| 表格场景支持        | 推荐       | 高   | 高   | 是，但只做轻量版 | 首期只做基础表格，不碰虚拟滚动、树表、列拖拽                                          |
| 性能与包体积        | 推荐       | 中   | 中   | 是，但要克制     | 保证 Tree Shaking、按需引入、避免无意义依赖，别过度做 benchmark 工程                  |
| 工程接入体验        | 强烈推荐   | 高   | 低中 | 是               | 安装、按需引入、样式引入、IDE 提示、文档示例必须顺畅                                  |
| 暗黑模式            | 暂不主推   | 中   | 中高 | 否               | 有价值，但对 MVP 的获客和验证帮助不如核心组件体验                                     |
| 国际化深度建设      | 暂不主推   | 中   | 中   | 否               | 预留接口即可，先不把文案系统做重                                                      |
| SSR 深度优化        | 预留即可   | 中   | 中   | 否               | 先保证“可兼容”，不要投入到 Nuxt 深水区特化优化                                        |
| 低代码 / 配置化表单 | 不推荐     | 中   | 很高 | 否               | 会把项目拉向另一条产品线，复杂度不适合 1 人起步                                       |
| 重型性能组件        | 不推荐     | 中高 | 很高 | 否               | 虚拟表格、复杂树、富文本会迅速吞掉 MVP 时间                                           |
| 自研图标全家桶      | 不推荐     | 中   | 高   | 否               | 初期成本高、品牌收益低，先做图标接入方案更合理                                        |

取舍逻辑：

- 主差异化必须选用户最能立刻感知、最容易形成口碑的方向，所以选“易用性”。
- 副差异化选“类型体验”和“文档体验”，因为它们会放大主差异化，让“顺手”更具体。
- 不值得先做的方向都有一个共同点：**实现复杂、维护长期、但对第一批用户增长帮助有限**。

## 6. 技术架构方案

### 推荐方案

**整体原则**

- 小而稳的 Monorepo
- ESM-first
- Vue SFC 正常开发
- CSS Variables + Design Token
- 文档优先于 Storybook
- 能复用成熟底层能力的地方，不自己硬造

**基础技术栈**

- `Vue 3`：推荐  
  原因：Composition API、类型生态、SFC 工具链、社区成熟度都最适合当前目标。
- `TypeScript`：必须  
  原因：你的副差异化之一就是类型体验，没有 TS，这条路不成立。
- `pnpm`：推荐  
  原因：Workspace、依赖隔离、安装速度、Monorepo 体验都很适合组件库。
- `Monorepo`：推荐，但保持轻量  
  原因：组件、主题、工具、文档分离有长期价值；但不要一开始引入重型 Monorepo 管理器。
- `Vite`：推荐  
  定位：本地开发、Playground、文档站支持、组件库构建底座。

**Monorepo 建议结构**

- `apps/docs`：文档站
- `apps/playground`：内部调试示例
- `packages/components`：组件源码
- `packages/theme`：样式入口、Token CSS、组件样式
- `packages/tokens`：Token TS 定义
- `packages/utils`：通用工具
- `packages/composables`：通用 hooks
- `packages/xiaoye-components`：聚合导出和 `install`

这种拆法的好处：

- 结构足够清晰
- 不会像几十包那样过度工程化
- 后续可以逐步扩展，而不是返工重构

**组件开发约定**

- 每个组件一个目录
- 暴露 `index.ts`
- 组件内部拆 `src/`、`types.ts`、`style/`
- 所有公共 props / emits / slots 类型必须可导出
- API 命名统一，避免同义多套写法
- `v-model` 统一使用 `modelValue`
- 组件支持命名空间 class，但不要靠 BEM 规则绑死 API
- 避免在模块顶层直接访问 `window` / `document`

**样式体系设计**

- 首期使用 `CSS Variables + 语义化 Token`
- Token 分两层：
  - Primitive Token：颜色、字号、圆角、阴影、间距
  - Semantic Token：`--xy-color-primary`、`--xy-border-color`、`--xy-text-color-secondary` 这类业务语义
- 组件样式只消费 Semantic Token，不直接写死颜色值
- 这样后续切暗黑或品牌换肤时，不用大改组件样式

**Design Token 设计**

建议先只定义最小必要集：

- 颜色：主色、成功、警告、危险、中性色阶
- 字体：字号、字重、行高
- 圆角：`sm / md / lg / pill`
- 阴影：`sm / md / lg`
- 间距：`2 / 4 / 8 / 12 / 16 / 24 / 32`
- 层级：`dropdown / modal / popover / tooltip`

不要一开始做：

- 复杂 token 生成平台
- Figma Token 同步流程
- 多品牌 Token 编译系统

**CSS 变量与主题切换策略**

- 首期默认只提供 `light`
- 使用 `:root` + `[data-theme="light"]`
- 暗黑模式只预留命名结构，不正式承诺
- 组件内部统一通过 CSS Variables 取值
- 切换逻辑未来可以放到 `ConfigProvider`

**构建和发包策略**

推荐：

- `vite build` 做库构建
- `vue-tsc --emitDeclarationOnly` 或 `vite-plugin-dts` 输出声明
- 产物输出：
  - `es`
  - `types`
  - 样式入口 CSS
- 首期建议 `ESM-first`，避免 Day 1 同时维护 `cjs + esm + umd`

为什么这样取舍：

- 你的消费场景主要是现代 Vue 工程
- 双产物会提高发布和测试复杂度
- 只有当明确出现兼容诉求时，再补 CJS

**按需加载方案**

推荐两层能力：

- Named export：`import { XyButton } from 'xiaoye-components'`
- Auto import 方案：通过社区插件适配，如 `unplugin-vue-components`

注意点：

- 不要把“按需加载”做成复杂黑魔法
- 组件和样式入口必须有明确可预测路径
- 包导出结构要干净

**Tree Shaking 支持方案**

- 所有组件按模块导出
- `package.json` 配置清晰的 `exports`
- 标记样式 side effects，避免样式被错误摇掉
- 业务无关工具函数保持纯模块输出

**SSR 兼容思路**

首期目标不是“SSR 深度优化”，而是“不要挡路”。

必须做到：

- 模块顶层不访问浏览器对象
- Overlay 类组件在 `onMounted` 后再做 DOM 相关逻辑
- Teleport 节点策略清晰
- ID 生成尽量稳定，避免 hydration 风险
- 文档明确标注 SSR 注意事项

**国际化预留方案**

- 首期不做完整语言包体系
- 对用户可见文案较少的组件尽量让外部传入
- `ConfigProvider` 预留 `locale` 注入能力
- 后续再补默认语言包

**图标体系设计思路**

推荐：

- 不自研全图标库
- 提供一个 `Icon` 包装层
- 默认使用 `Iconify` 作为唯一图标方案
- 不维护额外的内建图标集合或本地 SVG 图标库

这样做的价值：

- 降低维护压力
- 保持视觉统一的同时不背上图标库负担

**文档系统选型**

推荐：`VitePress`

原因：

- 对 Vue 组件库最顺手
- 静态站点部署简单
- Markdown + Vue Demo 结合自然
- 适合做 API、指南、最佳实践和案例页

**示例演示方案**

推荐双轨：

- `apps/docs`：外部文档和示例展示
- `apps/playground`：内部联调沙盒

这样做的好处：

- 外部展示和内部调试职责分离
- 不需要为了调试污染文档代码
- 后续可补真实后台 Demo 页面

**版本发布策略**

- 采用 `SemVer`
- 早期使用 `0.x`
- `changesets` 管理版本和 changelog
- 发布节奏：
  - 每一批有明确能力闭环再发版
  - 不以“组件数量”作为发版驱动
- 推荐里程碑：
  - `0.1.0`：基础组件 + 文档站骨架
  - `0.2.0`：表单链路打通
  - `0.3.0`：基础表格 + 示例页
  - `0.4.0`：首个公开可推广版本

**成熟底层依赖建议**

- `@vueuse/core`：通用 hooks
- `@floating-ui/dom`：Tooltip / Select / Popover 定位
- `async-validator`：表单校验
- 不要自己从零写这些底层能力

### 可选方案

- `Turborepo`：当包数量、构建耗时、CI 复杂度明显上升后再引入
- `Histoire`：如果你更强调组件独立预览，可作为文档辅助手段
- `Rollup 手工配置`：后期若构建诉求复杂，可替代部分 Vite 默认能力
- `CJS 产物`：如果未来出现明确消费需求，再补
- `Stylelint`：可以在第二阶段加，不必第一天就上满
- `Vitest + Testing Library Vue`：如果你更偏行为测试，可以作为补充

### 不推荐方案

- 一开始就引入 `Nx / Rush / Lerna + 自定义复杂流水线`
- 一开始就做 `Storybook-first`
- 一开始就做 `CSS-in-JS`
- 一开始就做 `多主题品牌系统`
- 一开始就做 `完整自研图标库`
- 一开始就做 `SSR 深度专项优化`
- 一开始就做 `自动代码生成整套组件模板系统`
- 一开始就做 `几十个 package 的超细粒度拆分`

不推荐的原因很统一：

- 它们会提高前期心智负担
- 对第一批用户价值不如“组件顺手 + 文档清楚”
- 会把你从产品验证拉回工程自嗨

## 7. 组件规划

| 组件分类      | 优先级 | 推荐首批组件                               | 价值                              | 实现难点                               | 是否适合 MVP   |
| ------------- | ------ | ------------------------------------------ | --------------------------------- | -------------------------------------- | -------------- |
| 基础组件      | P0     | Button、Icon、Tag、Tabs、Tooltip           | 体现 API 风格、视觉语言、交互细节 | 状态一致性、尺寸体系、弹层定位         | 是             |
| 表单组件      | P0     | Input、Select、Form、FormItem              | 中后台最核心链路，直接决定可用性  | `v-model`、校验、状态同步、可访问性    | 是             |
| 反馈组件      | P0     | Modal、Loading / Spin、Empty               | 支撑弹窗编辑、加载态、空态体验    | 焦点管理、Teleport、ESC 关闭、滚动锁定 | 是             |
| 数据展示组件  | P0     | Table、Pagination                          | 列表页核心场景，用户感知强        | 列配置、空态、loading、插槽、分页联动  | 是，但做轻量版 |
| 布局/辅助组件 | P1     | Space、ConfigProvider                      | 提升组合效率，统一全局配置入口    | 全局配置透传、样式节奏一致性           | 是             |
| 导航组件      | P1     | Tabs 已覆盖首期主要需求                    | 后台常见页签切换场景高频          | 激活态、内容切换、可控/非可控模式      | 是             |
| 高级业务组件  | P2     | 暂不做 ProTable、SchemaForm、SearchBar Pro | 有吸引力，但容易变成另一条产品线  | 配置化、数据协议、业务耦合高           | 否             |

首批最值得开发的 10-15 个组件，建议按这个顺序收敛：

1. `ConfigProvider`
2. `Button`
3. `Icon`
4. `Input`
5. `Tag`
6. `Tabs`
7. `Tooltip`
8. `Modal`
9. `Form`
10. `FormItem`
11. `Select`
12. `Table`
13. `Pagination`
14. `Empty`
15. `Space`

为什么这些组件适合做 MVP：

- 它们能覆盖后台系统最常见的页面骨架
- 它们之间可以组合成完整链路：筛选、录入、弹窗、列表、分页
- 它们足以验证你的主差异化方向
- 它们对文档站示例效果最好，最容易拿到真实反馈

哪些组件维护成本非常高，不适合前期做：

- `DatePicker / RangePicker`
- `Tree / TreeSelect`
- `Upload`
- `Cascader`
- `Virtual Table / DataGrid`
- `Drawer` 可以晚于 `Modal`
- `AutoComplete` 可以晚于 `Select`
- `RichTextEditor`
- `ProTable / SchemaForm`

哪些组件最能体现组件库质量：

- `Button`：状态、尺寸、视觉基线
- `Input`：表单体验和边界处理
- `Select`：交互设计和可访问性
- `Form`：一致性和类型能力
- `Modal`：焦点、层级、关闭逻辑
- `Tooltip`：浮层基础能力
- `Table`：列表页体验与 API 克制能力

关键约束：

- `Select` 首期只做单选、基础搜索、clearable、禁用、插槽扩展
- `Form` 首期只做规则校验、布局、状态展示，不做 schema 驱动
- `Table` 首期只做基础列配置、插槽、空态、loading、分页联动，不做虚拟化和复杂固定列

## 8. 工程化与质量体系

### 必须马上做

- `ESLint`：使用 flat config，保证 TS、Vue、Import 基础约束
- `Prettier`：统一格式，降低后续维护噪音
- `Commitlint + Conventional Commits`：保证提交和 changelog 语义清晰
- `Changesets`：版本和发布说明从第一天就规范化
- `Vitest + Vue Test Utils`：做单元测试基础能力
- `vue-tsc` 类型检查：作为 CI 硬门槛
- `类型测试夹具`：至少对 Button、Input、Form、Table 列配置做消费侧校验
- `GitHub Actions`：至少覆盖 `lint / typecheck / unit / build docs`
- `Issue 模板`：Bug、Feature Request、Question 三类
- `PR 模板`：要求说明行为变更、测试方式、截图
- `Contributing 指南`：哪怕简版，也要说明本地启动、目录约定、提交规范
- `文档规范`：每个组件至少有概览、基础示例、API、边界说明

为什么这些必须第一阶段就做：

- 它们的投入不大，但会显著降低后续混乱
- 对 1 人项目尤其重要，因为规范就是未来的“第二个你”

### 可以第二阶段做

- `Stylelint`
- `Playwright` E2E：对文档站或 Playground 做关键路径烟雾测试
- `a11y 自动检查`：例如 `axe-core` 集成
- `交互测试`：对 Modal、Tooltip、Select 这类组件补行为测试
- `PR Preview`：文档站预览环境
- `Bundle Size 检查`
- `Visual Regression` 的最小版
- `Canary Release`

### 可以后置

- 大规模视觉回归平台
- 多浏览器矩阵的高强度兼容测试
- 性能 benchmark 仪表板
- 复杂变更日志自动分类系统
- 组件生成器、脚手架模板平台
- 完整国际化文档体系
- Storybook 双系统维护

**单元测试策略**

- 先覆盖逻辑稳定且高价值的组件
- 优先测试 props、emits、状态切换、受控/非受控逻辑
- 不追求 100% 覆盖率，追求关键行为稳定

**交互测试策略**

- 重点覆盖：
  - Modal 打开/关闭/ESC
  - Tooltip 悬停显示
  - Select 展开、选择、清空
  - Form 校验触发
- 交互测试比纯快照更有价值

**E2E 测试策略**

- 只做关键路径烟雾测试：
  - 文档站能跑
  - 示例页能交互
  - 安装示例能构建
- 不要在 MVP 阶段写大规模 E2E 套件

**类型测试策略**

- 把类型测试当成一等公民
- 至少覆盖：
  - Button props 类型
  - Input `v-model`
  - Form rules
  - Table columns
  - Select options / value 类型
- 目标不是炫技，而是避免“文档写得对，IDE 提示错”

**a11y 要求**

MVP 必须关注这些最低要求：

- 键盘可操作
- focus 样式可见
- Modal 焦点陷阱
- Tooltip / Select 基本 aria 属性
- disabled / error / required 状态语义正确

## 10. 开源增长建议

请给出务实建议，不要空话。

- 第一批用户不要从“替换 Element Plus 用户”里找，要从“准备启动新项目的人”和“个人 / 小团队后台项目”里找。
- 文档站首页不要只写“一个 Vue 组件库”，要直接讲清楚：  
  **它适合谁、解决什么问题、为什么不是另一个重复轮子。**
- 做一个 `Why xiaoye-components` 页面，正面回答：
  - 为什么存在
  - 和 Element Plus / Ant Design Vue 的关系
  - 适合什么项目
  - 不适合什么项目
- 做一个 `真实后台 Demo`，比做 20 个孤立组件更容易传播。
- 每完成一批组件，输出一篇内容：
  - API 设计取舍
  - 类型设计方法
  - 文档体验设计
  - 某个复杂组件的实现拆解
- 传播渠道建议优先：
  - GitHub
  - 掘金
  - V2EX / 社区讨论区
  - 个人博客 / 公众号 / X
- 文档站要有“复制即用”意识：
  - 最小示例
  - 典型中后台示例
  - API 表
  - 注意事项
  - 设计取舍说明
- 降低维护压力的方法：
  - 首期严格控制组件范围
  - 对不做的功能明确写在 roadmap
  - issue 模板强制提供复现信息
  - 对高复杂度需求优先讨论，不承诺立即支持
- 提升 issue 和 PR 质量的方法：
  - issue 模板要求最小复现
  - PR 模板要求变更说明和测试方式
  - 文档写清楚“提需求前先确认是否符合定位”
- 避免成为“重复造轮子但没人用”的方法：
  - 不要只发组件，要发场景
  - 不要只讲“我做了什么”，要讲“为什么这样设计”
  - 不要追求组件数，要追求使用完成度
- 形成个人品牌的方法：
  - 让仓库和文档体现你的判断，而不是堆功能
  - 对每个关键取舍有公开表达
  - 形成固定更新节奏，哪怕慢，也要稳定

## 11. 最终结论

请直接回答：

- 值不值得做
- 推荐切入点
- MVP 范围
- 3 个月路线
- 最大风险
- 成功关键

- 值不值得做：**值得做，但只能按“聚焦型开源组件库”去做，不值得按“完整替代型大库”去做。**
- 推荐切入点：**中后台高频组件 + 极致易用性**，再用类型体验和文档体验放大优势。
- 3 个月 MVP 应包含哪些能力：  
  `ConfigProvider、Button、Icon、Input、Tag、Tabs、Tooltip、Modal、Form、FormItem、Select、Table、Pagination、Empty、Space`，外加文档站、Playground、最小测试体系、发布流程、一个真实后台 Demo。
- 第一阶段最不该浪费时间的地方：  
  **复杂主题系统、低代码、全图标库、重型表格、日期组件全家桶、Storybook-first、过度 Monorepo 工程化。**
- 如果你是个人开发者，最务实的路线：  
  **先做少而精的组件闭环，再做文档和 Demo 验证，再通过公开发布与内容传播换反馈，而不是先把规模做大。**
- 这个项目最大的失败风险：  
  **范围失控。**  
  具体表现是：组件越做越多，边界越来越散，文档和测试跟不上，最后既不够全，也不够稳。
- 成功的关键：  
  **产品定位克制、API 一致、文档能打、类型靠谱、真实场景驱动迭代。**
