# xiaoye-components v0 基础仓库落地说明

## 1. 文档目标

本文记录 `xiaoye-components` 从空仓库落地为可运行 `v0` 基础仓库的技术实现方案。

这不是开发流水账，而是一次针对以下目标的工程落地说明：

- 建立一个适合个人开发者长期维护的 Vue 组件库骨架
- 保证组件开发、类型检查、测试、文档、构建和发布链路从第一阶段就能跑通
- 在有限范围内做出一套可展示、可验证、可继续迭代的 MVP 基础能力

本文重点说明：

- 为什么这样设计
- 核心目录和模块如何协作
- 第一阶段落下了哪些工程能力
- 当前方案解决了什么问题，保留了哪些后续演进空间

## 2. 落地范围

本次落地不是“完整组件库开发”，而是建立一套可持续演进的基础仓库。

范围包括：

- Monorepo 目录结构
- 根级工程化配置
- 组件库构建与类型产物输出
- 主题系统与 Token 基础
- 文档站与 Playground
- 首批最小可用组件
- 测试、类型检查、Lint、构建验证链路
- Git 初始化与开源协作模板

范围之外的内容包括：

- 重型组件能力，如虚拟表格、树形组件、日期组件
- 多主题品牌系统和完整暗黑模式
- 低代码、Schema 表单、Pro 级业务组件
- 复杂 SSR 优化和多运行时兼容专项

这意味着当前仓库的定位是：

**一个可运行、可发布、可继续扩展的组件库底座，而不是一个已经完成的大而全组件库。**

## 3. 整体架构

### 3.1 目录结构

项目采用轻量 Monorepo 结构，核心目录如下：

```text
apps/
  docs/           # VitePress 文档站
  playground/     # 本地联调与场景验证
packages/
  components/     # 组件源码
  composables/    # 组件共用 hooks
  utils/          # 安装器、DOM 工具、通用类型
  tokens/         # 设计 Token 定义
  theme/          # CSS Variables 与组件样式
  xiaoye-components/  # 聚合导出与发布入口
tests/
  types/          # 消费侧类型测试
```

这个结构的取舍逻辑很明确：

- 不拆成几十个包，避免前期过度工程化
- 也不把所有内容堆到一个目录里，避免后续失去扩展能力
- `apps` 和 `packages` 分离，确保文档展示、内部调试和库本体职责清晰

### 3.2 模块分层

当前仓库采用自下而上的分层方式：

1. `tokens`
2. `theme`
3. `utils`
4. `composables`
5. `components`
6. `xiaoye-components`
7. `docs / playground`

各层职责如下：

- `tokens` 只负责设计 Token 的常量定义
- `theme` 负责 CSS Variables、基础 reset 和组件样式
- `utils` 负责安装器、DOM 工具、通用类型
- `composables` 负责组件共用逻辑，例如命名空间、全局配置、z-index
- `components` 负责组件本身
- `xiaoye-components` 负责统一安装、统一导出和最终发布入口
- `docs / playground` 负责验证与展示，而不是承载组件逻辑

这个分层的价值在于：

- 样式、逻辑、发布边界清楚
- 文档与实际包结构一致
- 后续可以演进，但不需要推翻第一阶段设计

### 4.2 为什么使用 Vite 8

当前根构建链路已升级到 `Vite 8.0.1`。

升级的原因不是单纯追新，而是出于两个现实考虑：

- 使用当前稳定版，避免基础仓库刚建立就落后于主构建生态
- 与后续 Vue 3 工程、Playground、本地调试体验保持一致

与之配套同步升级的还有：

- `@vitejs/plugin-vue 6.0.5`
- `vitest 4.1.0`

这样做避免了“主构建版本升级，但测试链路仍停留在旧 peer 范围”的问题。

### 4.3 类型配置的处理方式

项目将 TS 配置分成多层：

- `tsconfig/base.json`：公共编译选项和 alias
- `tsconfig/packages.json`：组件与库源码类型检查
- `tsconfig/apps.json`：文档和 Playground 类型检查
- `tests/types/tsconfig.json`：消费侧类型测试
- `tsconfig.build.json`：声明文件生成专用配置

这里有一个关键取舍：

**类型检查配置和声明文件生成配置分离。**

这样做是为了避免：

- `vite-plugin-dts` 错误地把测试文件、文档文件或应用文件拉进声明产物
- `rootDir` 与输出目录冲突，导致声明文件生成报错

这是第一阶段比较重要的工程稳定性处理。

## 5. 组件库构建与发布入口

### 5.1 发布入口

组件库发布入口位于：

- [packages/xiaoye-components/index.ts](/Users/xiaoye/Desktop/AI/自研组件库/xiaoye-components/packages/xiaoye-components/index.ts)
- [packages/xiaoye-components/package.json](/Users/xiaoye/Desktop/AI/自研组件库/xiaoye-components/packages/xiaoye-components/package.json)

发布入口承担三件事：

- 统一导出所有组件
- 提供 `install(app)`，支持整库注册
- 引入统一样式入口

当前产物为：

- `ESM` 构建产物
- 类型声明文件
- 样式入口 `style.css`

这是一个刻意收敛的策略。

第一阶段不同时维护 `cjs + esm + umd`，原因是：

- 维护复杂度明显增加
- 当前目标用户主要是现代 Vue 工程
- 对 1 人项目来说，先保证稳定比先扩大兼容范围更重要

### 5.2 样式与 sideEffects

在发布包中保留了样式 side effects 标记，避免样式在 Tree Shaking 时被错误移除。

这意味着：

- JS 可以被 Tree Shaking
- 样式不会因为过度摇树而丢失

对于组件库来说，这是比“理论上更纯粹”更务实的做法。

## 6. 主题系统与 Token 设计

### 6.1 Token 层

Token 定义位于：

- [packages/tokens/src/primitives.ts](/Users/xiaoye/Desktop/AI/自研组件库/xiaoye-components/packages/tokens/src/primitives.ts)
- [packages/tokens/src/semantic.ts](/Users/xiaoye/Desktop/AI/自研组件库/xiaoye-components/packages/tokens/src/semantic.ts)

当前分为两层：

- Primitive Token：基础色、圆角、字号、间距、阴影
- Semantic Token：主色、文本色、边框色、背景色、覆盖层等语义变量

这套设计的目的不是“追求设计系统完整度”，而是为后续主题扩展留出口。

组件样式直接消费语义变量，不直接写死视觉值，这样后续如果增加：

- 暗黑模式
- 品牌换肤
- 主题扩展

就不需要逐个重写组件样式。

### 8.1 组件选择原则

首批组件的选择遵循三个原则：

- 能覆盖中后台高频场景
- 能体现 API 风格和组件质量
- 维护复杂度在 1 人项目可控范围内

因此优先做的是：

- 输入与筛选：`Input`、`Select`
- 表单录入：`Form`、`FormItem`
- 弹层交互：`Modal`、`Tooltip`
- 数据展示：`Table`、`Pagination`
- 基础承托组件：`Button`、`Tag`、`Tabs`、`Space`、`Empty`

### 8.2 首期组件的边界控制

这些组件都做了范围收敛。

例如：

- `Select` 只做单选、基础搜索、清空和禁用
- `Form` 只做基础校验与状态联动，不做 schema 驱动
- `Table` 只做轻量列配置与插槽扩展，不做虚拟化和复杂固定列

这样做的根本原因是：

**首期目标是验证产品方向和工程基础，而不是用组件数量制造完成度假象。**

## 11. 实施过程中解决的关键问题

### 11.1 Vite 8 升级同步问题

根构建升级到 `Vite 8` 后，如果不同时调整：

- `@vitejs/plugin-vue`
- `vitest`

就容易出现 peer 范围不一致的问题。

因此本次升级采用的是整条构建链路同步调整，而不是单包升级。

### 11.2 声明文件生成与 rootDir 冲突

初始阶段，声明文件生成出现过 TS `rootDir` 相关错误。

根因是：

- 类型检查配置与构建产物配置没有清晰分离
- `vite-plugin-dts` 把不该纳入产物的文件拉进了声明生成过程

最终通过单独引入 [tsconfig.build.json](/Users/xiaoye/Desktop/AI/自研组件库/xiaoye-components/tsconfig.build.json) 解决，确保声明产物只覆盖真正的发布源码。

### 11.3 pnpm Workspace 下的 VitePress 软链问题

在 `pnpm workspace` 环境下，`VitePress` 构建阶段会处理 `vue` 的软链目录。

当前仓库通过在文档脚本中先清理 `apps/docs/node_modules/vue` 的方式，规避了重复软链导致的 `EEXIST` 问题。

这个处理位于：

- [apps/docs/package.json](/Users/xiaoye/Desktop/AI/自研组件库/xiaoye-components/apps/docs/package.json)

这属于工程兼容层面的稳定性修正，不影响组件 API，但保证了文档链路可用。
